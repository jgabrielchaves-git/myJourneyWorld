# Journey World
# Development - MVP Design Principles

**Versão:** 1.0
**Status:** Living Document

---

# Objetivo

Este documento define os princípios para desenvolvimento do MVP do Journey World.

Seu propósito é manter o projeto simples, sustentável e focado em validar as principais hipóteses do produto antes de expandir funcionalidades.

Sempre que houver dúvida entre construir algo mais sofisticado ou mais simples, este documento deve orientar a decisão.

---

# Objetivo do MVP

O MVP não existe para provar que conseguimos desenvolver um jogo.

Ele existe para responder uma única pergunta:

"A criança sente vontade de voltar amanhã?"

Todo o restante pode ser evoluído posteriormente.

---

# Hipóteses que queremos validar

O MVP deve validar principalmente:

- A criança sente prazer em cultivar sua jornada.
- As missões incentivam prática diária.
- As estrelas geram senso de conquista.
- A Casa cria sentimento de pertencimento.
- O Jardim desperta interesse pelas diferentes jornadas.
- A Árvore representa visualmente o crescimento.
- Os pais participam da jornada.
- O Storybook gera valor emocional.

Qualquer funcionalidade que não ajude a validar uma dessas hipóteses deve ser reavaliada.

---

# Filosofia

Construir pequeno.

Validar cedo.

Aprender rápido.

Evoluir continuamente.

---

# Simplicidade acima de sofisticação

Sempre preferir a solução mais simples que comunique a ideia.

Exemplos:

✔ Trocar uma imagem.

❌ Criar uma animação complexa.

---

✔ Trocar de tela.

❌ Criar movimentação do personagem.

---

✔ Mostrar uma árvore em outro estágio.

❌ Animar o crescimento da árvore.

---

# Reutilização

Sempre reutilizar componentes.

Evitar criar sistemas específicos quando um sistema existente pode atender a necessidade.

---

# Fantasia

A fantasia deve enriquecer a experiência.

Nunca aumentar a complexidade do desenvolvimento.

Nunca dificultar a navegação.

---

# Clareza

Toda tela deve responder apenas uma pergunta.

Exemplos:

Casa

"O que quero fazer hoje?"

---

Jardim

"Qual jornada quero cultivar?"

---

Reino

"O que preciso aprender aqui?"

---

Missão

"O que devo fazer agora?"

---

Conclusão

"O que conquistei?"

---

# Navegação

O jogador nunca deve se sentir perdido.

Poucos caminhos.

Poucas decisões.

Poucos cliques.

---

# Feedback

Toda ação importante deve gerar um retorno imediato.

Exemplos:

- estrela conquistada;
- árvore evoluiu;
- missão concluída;
- nova carta disponível.

Não é necessário utilizar animações complexas.

Mudanças visuais simples são suficientes.

---

# Escopo

Toda nova ideia deve ser classificada.

## MVP

Necessária para validar a experiência.

Implementar agora.

---

## V2

Melhora significativamente a experiência.

Implementar após validação do MVP.

---

## Visão

Ideia estratégica para longo prazo.

Documentar no backlog.

Não implementar neste momento.

---

# Critérios para adicionar uma funcionalidade

Antes de iniciar qualquer desenvolvimento, responder:

Esta funcionalidade valida uma hipótese importante?

Ela melhora significativamente a experiência do MVP?

Ela pode ser implementada com simplicidade?

Ela reutiliza componentes existentes?

Ela aumenta ou reduz a complexidade do projeto?

Se a maioria das respostas for negativa, a funcionalidade deve permanecer no backlog.

---

# Critérios de sucesso do MVP

O MVP será considerado bem-sucedido se:

- a criança desejar voltar espontaneamente;
- os pais perceberem valor na experiência;
- a prática se tornar mais consistente;
- a jornada fizer sentido sem explicações complexas;
- o sistema conseguir registrar essa evolução.

---

# Não perseguimos perfeição

O MVP não precisa possuir:

- todas as jornadas;
- todas as animações;
- todos os sistemas;
- todos os cosméticos;
- todos os eventos.

Ele precisa apenas entregar uma experiência suficientemente boa para validar o conceito.

---

# Princípios

Cada linha de código deve aproximar o projeto de uma validação real.

Cada tela deve existir por um motivo.

Cada sistema deve resolver um problema.

Cada nova funcionalidade deve justificar sua própria existência.

Sempre que houver dúvida, escolher a solução mais simples capaz de encantar a criança.
